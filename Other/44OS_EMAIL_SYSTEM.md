## Application contract

- Repository templates are the source of truth. Each outbox row stores a stable template key, template version, event key, recipient, and minimal render payload. HTML and plain text are rendered immediately before delivery; message bodies and secrets are not stored in provider-event history.
- `event_key` is globally unique and is also sent to Resend as the API idempotency key. Database claiming prevents concurrent workers, and the provider key protects a retry whose first response was lost. Resend retains provider idempotency keys for 24 hours, so ambiguous failures retry only inside a conservative 23-hour window. After that, the event requires provider reconciliation and a deliberate operator decision; it is never blindly resent after the provider key expires. Source: [Resend idempotency keys](https://resend.com/docs/dashboard/emails/idempotency-keys).
- Delivery is fail-closed behind `email_delivery_controls`. Credentials alone cannot turn it on. The worker is server-only and accepts an independent cron secret; no browser can enqueue arbitrary messages or invoke Resend.
- Welcome is keyed once per account. Purchase confirmation is keyed to the verified paid order transition. Refund/cancellation is keyed to the authoritative refund/provider event. Fulfillment/tracking is keyed to the authoritative Printful event. Support acknowledgement is keyed once per support case.
- Provider delivery, bounce, and complaint webhooks require raw-body signature verification and idempotent event storage. Resend webhooks are at-least-once and may arrive out of order, so `svix-id` is the durable duplicate boundary. They retain identifiers, timestamps, event type, and sanitized failure classification—not message content or secrets. Hard bounce and complaint suppress future application sends to the address; operational suppression is reviewable and newsletter consent remains separate. Source: [Resend webhook delivery guarantees](https://resend.com/docs/webhooks/introduction).

## Newsletter and support boundaries

- Newsletter consent is explicit, timestamped, source-labelled, and independently revocable. Signup, purchase, or opening a support case never implies consent.
- The database consent ledger remains authoritative. Resend Contact/Topic synchronization is an asynchronous outbox operation, and Broadcasts may target only the newsletter Topic after reconciliation. The Topic is default opt-out because Resend defines that mode as requiring an explicit subscription before a Contact receives the mail; Resend-managed unsubscribe events must update the local revocation state. Sources: [Resend Topics](https://resend.com/docs/dashboard/topics/introduction), [contact.updated](https://resend.com/docs/webhooks/contacts/updated).
- `support@44os.com` must be a real monitored mailbox, forwarding address, or shared-mailbox workflow before support intake is activated. Durable cases record requester identity, status, priority, assignment, reply owner, and append-only public/internal history. Application messages do not impersonate inbound email receiving.
- A human reply is sent from the monitored mailbox. The application stores reply ownership/history and may send acknowledgement/status notifications through Resend, but it does not enable Resend Receiving on root `44os.com`.
